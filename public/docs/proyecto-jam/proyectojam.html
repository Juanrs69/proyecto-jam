1. Nombre del Proyecto

 Visita Segura

 2. Descripción General del proyecto

El sistema permite gestionar el ingreso y salida de visitantes a una institución o empresa, autorizaciones,generar reportes y controlar el acceso de forma segura y organizada.

 3. Objetivos

- Gestionar de forma digital los accesos de visitantes

- Mejorar la seguridad y trazabilidad dentro de la institución.

- Proporcionar un historial de accesos por fecha, persona y ubicación.

Objetivo del análisis

La fase de análisis constituye la base del ciclo de vida del software. Su propósito es identificar con claridad qué debe hacer el sistema y cuáles son las expectativas de los usuarios, antes de pasar a la etapa de diseño o codificación. Un análisis mal ejecutado puede comprometer todo el proyecto.

. Alcance del Proyecto

1 Alcance incluido (funcionalidades y detalle técnico)

1.1Registro completo de visitantes

Descripción: Formulario para crear visitantes con: nombre completo, documento (único), teléfono (opcional), correo (opcional), motivo de la visita y persona a visitar.

Criterio de aceptación: Al enviar el formulario se crea un registro en la tabla visitantes. El documento debe ser único; si existe, se reutiliza el registro. Validaciones en frontend y backend.

1.2Notificación automática al empleado receptor

Descripción:Cuando se registra una visita, el sistema crea una notificación y (configurable) envía una notificación interna (registro en tabla notificaciones) y opcionalmente correo electrónico.

Criterio de aceptación: Al registrar la visita se inserta una fila en notificaciones. Si el sistema tiene SMTP configurado, envía correo y registra el resultado en la tabla.

1.3Autorización o rechazo por parte del empleado

Descripción: Panel para que el empleado vea visitas pendientes y pueda autorizar o rechazar; acciones quedan registradas con marca temporal y usuario.

Criterio de aceptación: La acción cambia el campo estado en visitas a AUTORIZADA/RECHAZADA y se registra fecha de ingreso cuando se autoriza.

1.4Registro de hora de entrada y salida

Descripción: El sistema permite registrar la hora de entrada al autorizar y la hora de salida al finalizar la visita; trazabilidad completa por visita.

Criterio de aceptación: fecha ingreso y fecha salida aparecen en el registro de visitas y se muestran en la vista detalle.

1.5Búsqueda y filtrado de visitas

Descripción: Buscador por nombre, documento, empleado, fecha o rango de fechas; paginación para listados grandes.

Criterio de aceptación: Las consultas filtran la lista y la paginación funciona; operaciones con tiempos de respuesta < 2s (RNF02).

1.6Generación de reportes exportables (PDF y CSV)

Descripción: Generar reportes por rango de fechas, empleado o estado; exportar a CSV y a PDF (formato presentable).

Criterio de aceptación: El botón “Exportar” descarga archivo válido y legible; PDF con encabezado y filtros aplicados.

1.7Gestión de usuarios (CRUD)

Descripción: Administrador puede crear/editar/eliminar usuarios con roles (Recepcionista, Empleado, Administrador). Control de acceso por roles.

Criterio de aceptación: Solo administradores ven panel de usuarios; roles aplican permisos y validaciones.

1.8Interfaz web responsive

Descripción: Frontend responsive basado en Tailwind CSS y Alpine.js; accesible desde navegadores modernos.

Criterio de aceptación: Interface usable en desktop/tablet/móvil; pruebas manuales básicas en Chrome/Edge/Firefox.

2. Alcance excluido (límites explícitos)

2.1No incluye control biométrico (huella, reconocimiento facial).

2.2No incluye integración con sistemas de seguridad física (cámaras, cerraduras).

2.3No incluye app móvil nativa (solo web responsive).

2.4No incluye facturación o cobros.

2.1 Flujo principal (UC01 – Registrar visitante)

  1. Recepcionista accede a formulario “Registrar visitante”.
  2. Ingresa datos: nombre, documento, motivo, persona a visitar.
  3. El sistema valida datos obligatorios.
  4. Se crea un registro en la BD con estado = “pendiente”.
  5. Sistema invoca UC02 para notificar al empleado.

 

2.2 Flujo alterno (UC01.a – Datos incompletos)

  • Si faltan campos obligatorios, el sistema muestra mensaje de error y retorna al paso 2.
